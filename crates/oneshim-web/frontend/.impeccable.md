# ONESHIM Design Context

## Users

**Knowledge workers** (developers, designers, PMs, analysts) who spend 6-10+ hours daily at a desktop. ONESHIM runs as a background agent that monitors context (active windows, system metrics, screen captures) and surfaces AI-powered suggestions, coaching, and focus analytics.

Users interact with the product through:
- **Web Dashboard** (localhost:10090) â€” IDE-style shell for reviewing metrics, focus, reports, coaching history
- **MagicOverlay** â€” floating Tauri WebView for inline AI suggestions and GUI element detection
- **Control Box / Tracking Panel** â€” compact always-on status indicators
- **System Tray** â€” quick access (native, not WebView)

**Context of use**: Users are in the middle of focused work. The UI must never interrupt flow. Information density matters â€” knowledge workers want data-rich views, not empty marketing screens. The product should feel like a trusted instrument panel, not a chatbot toy.

**Emotional goals**: Confidence (my work is being tracked accurately), Calm (non-intrusive, always there when needed), Intelligence (the system understands my patterns).

## Brand Personality

**Intelligent. Precise. Composed.**

- **Intelligent**: The product analyzes behavioral patterns, predicts focus regimes, and coaches proactively. The UI should reflect analytical competence â€” clean data visualization, meaningful metrics, purposeful hierarchy.
- **Precise**: Every pixel should feel intentional. No decorative fluff. Spacing, alignment, and typography are tight and systematic.
- **Composed**: Calm under complexity. Even when displaying dense data (heatmaps, process lists, coaching goals), the interface remains visually quiet and organized.

## Aesthetic Direction

**Visual tone**: Technical-professional, IDE-adjacent. Inspired by developer tooling aesthetics â€” monospace accents for data, compact information density, semantic color over decorative color.

**References**:
- OpenAI Codex (desktop app) â€” minimum quality bar for polish, spacing, and interaction design
- Linear â€” information density + visual calm
- VS Code â€” shell structure (activity bar, side panel, status bar) already adopted
- Raycast â€” overlay interaction patterns, compact floating UI

**Anti-references**:
- Consumer SaaS dashboards with excessive whitespace and illustration-heavy empty states
- Notion-style soft/playful aesthetics â€” too casual for the analytical nature of this product
- Rainbow accent colors â€” brand teal only for primary actions/metrics (per existing feedback)

**Theme**: Light and Dark mode, both fully supported via CSS custom properties. No `dark:` Tailwind prefix anywhere.

**Typography**: System font stack (`-apple-system, BlinkMacSystemFont, "Segoe UI", Roboto`). No custom web fonts. Monospace for technical/numeric values.

## Design System (Existing)

ONESHIM has a mature, enforced design token system. **All design work must use existing tokens.**

### Token Architecture

| Layer | Location | Purpose |
|-------|----------|---------|
| CSS Custom Properties | `src/index.css` | RGB triplets for light/dark switching |
| Tailwind Extensions | `tailwind.config.js` | Semantic color classes (`bg-surface-base`, `text-content`) |
| JS Token Objects | `src/styles/tokens.ts` | Composable class strings for components |
| Variant Objects | `src/styles/variants.ts` | Component-specific variant maps |
| Lint Gate | `scripts/lint-design-tokens.sh` | CI rejects hardcoded colors/spacing |

### Color System

- **Brand**: Teal â€” `brand`, `brand-hover`, `brand-text`, `brand-signal`, `brand-bar`
- **Surface**: `base`, `elevated`, `muted`, `inset`, `sunken`, `overlay`

## Design Principles

1. **Token-first, always** â€” Import from `tokens.ts`. Never hardcode `text-slate-500` or `bg-[#fff]`. CI lint enforces this.
2. **Information density over whitespace** â€” Knowledge workers want data-rich views. Compact layouts, meaningful spacing, no decorative padding.
3. **Brand teal for meaning, not decoration** â€” Teal signals primary actions, active states, and key metrics. No rainbow accents. Semantic tokens (`success`/`warning`/`error`) for feedback only.
4. **Quiet confidence** â€” Animations are subtle and functional (150-300ms). No bounces, no attention-seeking effects. Motion serves state transitions, not decoration.
5. **Accessibility by default** â€” `focus-visible` ring on all interactive elements. WCAG AA contrast minimum. `addon-a11y` in Storybook. Semantic HTML structure.

## Technical Constraints

- **No custom fonts** â€” system font stack only (performance + native feel)
- **No `dark:` prefix** â€” all theming via CSS custom properties
- **No arbitrary Tailwind values** â€” spacing/color must come from token scale
- **Component stories required** â€” every component needs `*.stories.tsx`
- **Biome linting** â€” enforced via `pnpm lint`
- **Tailwind 3.4** (not v4) â€” class-based dark mode
