# D13 v2 Roadmap — Per-Domain RPCs, Streaming, TLS+Auth

**Date**: 2026-04-21
**Parent**: D13 v1 shipped as [PR #455](https://github.com/pseudotop/oneshim-client/pull/455) (proto), [PR #456](https://github.com/pseudotop/oneshim-client/pull/456) (impl), [PR #462](https://github.com/pseudotop/oneshim-client/pull/462) (integration test), [PR #463](https://github.com/pseudotop/oneshim-client/pull/463) (config field).
**Status**: Roadmap — implementation deferred to dedicated feature session(s)

## Context

D13 v1 exposes a minimal gRPC dashboard API with only `GetAgentInfo` + `HealthCheck`. Real value comes from per-domain RPCs that replicate the REST dashboard surface (frames, events, sessions, focus, metrics) for external tooling. v2 adds three orthogonal feature tracks: per-domain RPCs, streaming, and TLS+auth.

This doc captures the full v2 scope at the depth needed to execute in a dedicated feature session without re-discovering the proto schema choices, port wiring requirements, or test infrastructure patterns.

## Summary of sub-features

| Sub-feature | Effort | Risk | Dependency |
|-------------|--------|------|------------|
| **V2a. Per-domain RPCs** | ~2 days | Medium (port wiring) | None — extends DashboardServiceImpl |
| **V2b. Streaming RPCs** | ~2-3 days | Medium-High (backpressure, cancellation) | V2a (per-domain types reused) |
| **V2c. TLS + auth** | ~1-2 days | Medium (cert rotation, token storage) | None — wraps transport |

Recommended order: **V2a → V2b → V2c**. V2a establishes the port-wiring pattern; V2b reuses the types; V2c wraps independently.

## V2a. Per-Domain RPCs

### Goal
Add read-side RPCs covering the highest-value REST dashboard endpoints. The target set:

- `GetSessionStats(GetSessionStatsRequest)` → `SessionStatsResponse`
  Mirrors REST `GET /api/sessions/stats` — total session count, avg duration, focus ratio, top apps.
- `GetProductivityMetrics(GetProductivityMetricsRequest)` → `ProductivityMetricsResponse`
  Mirrors REST `GET /api/metrics/productivity` — CPU/memory/activity-per-hour rollups.
- `GetRecentFrames(GetRecentFramesRequest)` → `RecentFramesResponse`
  Mirrors REST `GET /api/frames?limit=N&since=...` — recent frame metadata (no image bytes).
- `GetFocusStats(GetFocusStatsRequest)` → `FocusStatsResponse`
  Mirrors REST `GET /api/focus/stats` — focus session count, avg minutes, distraction rate.

Four RPCs for v2a. Each is a thin wrapper around an existing REST handler's underlying service call — no new business logic.

### Proto schema additions

`api/proto/oneshim/dashboard/v1/dashboard.proto`:

```proto
service DashboardService {
  // ... existing GetAgentInfo + HealthCheck ...

  // Session domain
  rpc GetSessionStats(GetSessionStatsRequest) returns (SessionStatsResponse);
  rpc GetRecentFrames(GetRecentFramesRequest) returns (RecentFramesResponse);

  // Productivity domain
  rpc GetProductivityMetrics(GetProductivityMetricsRequest) returns (ProductivityMetricsResponse);
  rpc GetFocusStats(GetFocusStatsRequest) returns (FocusStatsResponse);
}

message GetSessionStatsRequest {
  // Time window in ISO-8601 RFC 3339 format. Both optional — defaults: now-7d → now.
  string from = 1;
  string to = 2;
}

message SessionStatsResponse {
  int64 total_sessions = 1;
  double avg_duration_minutes = 2;
  double focus_ratio = 3;
  repeated TopApp top_apps = 4;

  message TopApp {
    string app_name = 1;
    int64 session_count = 2;
    double total_minutes = 3;
  }
}

// GetRecentFrames / RecentFramesResponse
message GetRecentFramesRequest {
  // Defaults: limit=50, since=now-1h.
  uint32 limit = 1;
  string since = 2;
}

message RecentFramesResponse {
  repeated FrameMetadata frames = 1;

  message FrameMetadata {
    int64 frame_id = 1;
    string captured_at = 2;
    string trigger_type = 3;
    string app_name = 4;
    string window_title = 5;
    float importance = 6;
    // Ocr text NOT included — requires privacy sanitization path.
  }
}

// GetProductivityMetrics / ProductivityMetricsResponse
message GetProductivityMetricsRequest {
  string from = 1;
  string to = 2;
  // Bucket size in minutes. 0 or unset → 60 (hourly rollups).
  uint32 bucket_minutes = 3;
}

message ProductivityMetricsResponse {
  repeated MetricBucket buckets = 1;

  message MetricBucket {
    string start = 1;  // RFC 3339
    double cpu_avg_pct = 2;
    double memory_avg_mb = 3;
    uint32 active_keystrokes = 4;
    uint32 active_mouse_clicks = 5;
  }
}

// GetFocusStats / FocusStatsResponse
message GetFocusStatsRequest {
  string from = 1;
  string to = 2;
}

message FocusStatsResponse {
  int64 total_focus_sessions = 1;
  double avg_minutes_per_session = 2;
  double distraction_rate = 3;
  uint32 longest_session_minutes = 4;
}
```

### Port wiring

The `DashboardServiceImpl` struct currently only holds `started_at: Instant`. It needs read access to:

- **Sessions data**: `Arc<dyn SessionsReader>` — new port in `oneshim-core::ports::sessions_reader` OR reuse existing `WebStorage` trait (it has `sessions()` methods).
- **Metrics data**: `Arc<dyn MetricsReader>` — likely `WebStorage::metrics()`.
- **Frames data**: `Arc<dyn FrameStorage>` — already exists.
- **Focus data**: `Arc<dyn FocusStorage>` — already exists.

**Recommendation: reuse `Arc<dyn WebStorage>`** as the single dependency since it already aggregates all read APIs. DashboardServiceImpl becomes:

```rust
pub struct DashboardServiceImpl {
    started_at: Instant,
    storage: Arc<dyn WebStorage>,
}
```

`serve()` / `serve_optional()` signatures gain a `storage: Arc<dyn WebStorage>` parameter. Consumers (`src-tauri/src/app_runtime_launch.rs`) pass `sqlite_storage.clone() as Arc<dyn WebStorage>` at the existing spawn site.

### Implementation pattern (per RPC)

```rust
async fn get_session_stats(
    &self,
    req: Request<GetSessionStatsRequest>,
) -> Result<Response<SessionStatsResponse>, Status> {
    let req = req.into_inner();
    let from = parse_rfc3339_or_default(&req.from, /* default: now - 7d */);
    let to = parse_rfc3339_or_default(&req.to, /* default: now */);

    let stats = self.storage
        .session_stats(from, to)
        .await
        .map_err(|e| Status::internal(format!("session_stats: {e}")))?;

    Ok(Response::new(SessionStatsResponse {
        total_sessions: stats.total_sessions,
        avg_duration_minutes: stats.avg_duration_minutes,
        focus_ratio: stats.focus_ratio,
        top_apps: stats.top_apps.into_iter().map(|a| session_stats_response::TopApp {
            app_name: a.app_name,
            session_count: a.count,
            total_minutes: a.total_minutes,
        }).collect(),
    }))
}
```

Wrap each with `rpc_error_map!(e)` → `tonic::Status` helper if one doesn't exist; otherwise inline `map_err`.

### Tests

Extend `crates/oneshim-web/tests/grpc_dashboard_integration.rs` with 4 new tests:

- `grpc_dashboard_session_stats_end_to_end` — seed in-memory SqliteStorage, call GetSessionStats, assert `total_sessions > 0`
- `grpc_dashboard_recent_frames_end_to_end`
- `grpc_dashboard_productivity_metrics_end_to_end`
- `grpc_dashboard_focus_stats_end_to_end`

Each ~50 LoC. Follows the existing integration test pattern (ephemeral port + tonic client).

### Acceptance

- 4 new RPCs callable end-to-end through tonic client
- Responses match REST endpoint shape semantically (values identical when querying same data)
- `cargo test -p oneshim-web --features grpc-dashboard` — 4 new tests pass
- Proto regeneration via `./scripts/regenerate-dashboard-protos.sh` is idempotent

### Risks

- **`WebStorage` trait shape may not have session_stats/focus_stats** as direct methods — need to audit. If missing, either extend the trait (clean) or add a new `DashboardReader` super-trait aggregating read methods (cleaner separation).
- **`parse_rfc3339_or_default`** helper needs time-mocking in tests. Existing chrono-based utilities in `oneshim-core::time` may cover.

## V2b. Streaming RPCs

### Goal
Convert V2a's polling RPCs into streaming for live dashboards. Add 2 new streaming RPCs + keep V2a's unary RPCs for snapshots.

- `SubscribeMetrics(SubscribeMetricsRequest)` → **stream** `MetricBucket`
  Emits a `MetricBucket` every `bucket_interval_secs` (default 60). Respects client cancellation.
- `SubscribeEvents(SubscribeEventsRequest)` → **stream** `DashboardEvent`
  Emits a `DashboardEvent` on every new session/focus/frame in the underlying stream. Basically a tail of the real-time event broadcast.

### Proto additions

```proto
service DashboardService {
  // ... existing ...

  rpc SubscribeMetrics(SubscribeMetricsRequest) returns (stream MetricBucket);
  rpc SubscribeEvents(SubscribeEventsRequest) returns (stream DashboardEvent);
}

message SubscribeMetricsRequest {
  uint32 bucket_interval_secs = 1; // 0 → default 60
}

// Reuse ProductivityMetricsResponse.MetricBucket — extract to top-level.

message SubscribeEventsRequest {
  // Filter by event type: "session_start" | "session_end" | "focus_start"
  // | "focus_end" | "frame_captured". Empty = all.
  repeated string event_types = 1;
}

message DashboardEvent {
  string event_type = 1;
  string occurred_at = 2;
  // JSON-encoded payload specific to event_type. Gives forward-compat.
  string payload_json = 3;
}
```

### Implementation

Streaming in tonic uses `Pin<Box<dyn Stream<Item = Result<T, Status>> + Send>>`. Standard pattern:

```rust
type SubscribeMetricsStream = Pin<Box<dyn Stream<Item = Result<MetricBucket, Status>> + Send>>;

async fn subscribe_metrics(
    &self,
    req: Request<SubscribeMetricsRequest>,
) -> Result<Response<Self::SubscribeMetricsStream>, Status> {
    let interval_secs = req.into_inner().bucket_interval_secs.max(1);
    let storage = self.storage.clone();

    let stream = async_stream::stream! {
        let mut interval = tokio::time::interval(Duration::from_secs(interval_secs as u64));
        loop {
            interval.tick().await;
            match storage.metrics_snapshot().await {
                Ok(snap) => yield Ok(snap.into_proto()),
                Err(e) => yield Err(Status::internal(format!("metrics: {e}"))),
            }
        }
    };

    Ok(Response::new(Box::pin(stream)))
}
```

For `SubscribeEvents`, subscribe to `AppState.event_tx.subscribe()` (broadcast channel) and forward matching events.

### Backpressure / cancellation

- tonic handles cancellation when the client drops the stream — the async_stream generator is dropped, cleanup automatic.
- No explicit backpressure needed at generator level (tonic buffers per-connection).
- **Event drops** on `SubscribeEvents` need handling: `broadcast::Receiver::recv()` returns `Err(RecvError::Lagged)` when the client is slow. Send a sentinel `DashboardEvent{event_type: "lagged"}` and log.

### Tests

- `grpc_dashboard_subscribe_metrics_emits_buckets` — subscribe with interval=1s, collect 3 buckets, assert timestamps are ~1s apart.
- `grpc_dashboard_subscribe_events_filters_by_type` — emit 3 sessions + 2 frames, subscribe with filter=[session_start], expect 3 events.
- `grpc_dashboard_subscribe_events_handles_lag` — subscribe, flood 10000 events, verify client receives "lagged" sentinel and continues.

### Risks

- **tonic-prost stream type generation** — `build_client(true)` must include streaming support. Confirm the regenerate script output includes `SubscribeMetricsStream` type alias.
- **Test flakiness on time-based streams** — use `tokio::time::pause()` + `advance()` instead of wall-clock sleeps.
- **broadcast::Receiver capacity tuning** — existing `EVENT_CHANNEL_CAPACITY = 256` may be too tight for external subscribers. Bump to 4096 or add per-client buffering.

## V2c. TLS + Auth

### Goal
Protect the dashboard gRPC server when deployed beyond localhost. v1 is localhost-only (bind `127.0.0.1`); v2c adds optional TLS with a self-signed cert + shared-secret token auth.

### Design

- **TLS**: self-signed cert generated at first boot, stored in `{data_dir}/dashboard/cert.pem` + `key.pem`. Regenerated if either missing. External clients bypass verification with a fingerprint pin (documented in setup).
- **Auth**: shared-secret token in `Authorization: Bearer <token>` header. Token generated at first boot to `{data_dir}/dashboard/token.txt` (0600 mode). Authentication interceptor validates token before dispatching to handler.

Both are opt-in via config:

```rust
// crates/oneshim-core/src/config/sections/network.rs, existing WebConfig:
pub struct WebConfig {
    // ... existing fields ...
    /// D13-v2c: TLS for gRPC dashboard server.
    #[serde(default)]
    pub grpc_tls_enabled: bool,
    /// D13-v2c: require token auth for gRPC dashboard.
    #[serde(default)]
    pub grpc_auth_enabled: bool,
    /// D13-v2c: bind to 0.0.0.0 instead of 127.0.0.1. Requires TLS + auth.
    #[serde(default)]
    pub grpc_allow_external: bool,
}
```

### Implementation

TLS via `tonic::transport::ServerTlsConfig::new().identity(Identity::from_pem(...))`. Auth via `tonic::service::interceptor::interceptor_layer`.

```rust
let token = load_or_generate_token(&data_dir)?;
let auth_interceptor = move |req: Request<()>| -> Result<Request<()>, Status> {
    let header = req.metadata().get("authorization")
        .and_then(|v| v.to_str().ok())
        .unwrap_or("");
    if !header.starts_with("Bearer ") || &header[7..] != token {
        return Err(Status::unauthenticated("missing or invalid token"));
    }
    Ok(req)
};

let server = Server::builder();
let server = if config.grpc_tls_enabled {
    let identity = load_or_generate_identity(&data_dir)?;
    server.tls_config(ServerTlsConfig::new().identity(identity))?
} else {
    server
};
let service = InterceptedService::new(DashboardServiceServer::new(impl_), auth_interceptor);
server.add_service(service).serve(addr).await
```

### Tests

- `grpc_dashboard_auth_rejects_no_token` — connect without Authorization, expect `Status::Unauthenticated`.
- `grpc_dashboard_auth_accepts_valid_token` — connect with correct token, expect success.
- `grpc_dashboard_tls_serves_over_self_signed` — connect via TLS with fingerprint pin, expect success.
- `grpc_dashboard_external_binding_requires_tls` — config with `grpc_allow_external=true` + `grpc_tls_enabled=false` fails startup.

### Risks

- **Self-signed cert rotation** — no rotation in v2c. If the cert is compromised, user deletes the files and restarts. Document this.
- **Token leakage** — the token file is 0600 but an attacker with filesystem read has it. Acceptable for dev/local; production deployments should use OS keychain (`oneshim-storage::keychain`).
- **Cross-platform file perms** — Windows ACLs vs. Unix chmod. Use `file_secret_store` patterns from oneshim-storage.

## Execution recommendations

For a dedicated feature session:

### Session 1 (V2a, ~2 days)

1. Proto schema PR (message shapes + `service` entries). No impl yet.
2. Regenerate protos with `build_client(true)`.
3. Extend `DashboardServiceImpl` with `Arc<dyn WebStorage>`, wire 4 RPCs one at a time (TDD — test first).
4. Integration tests per RPC.
5. Update the main binary spawn site to pass storage.

### Session 2 (V2b, ~2-3 days)

6. Add streaming RPCs to proto + regenerate.
7. Implement `SubscribeMetrics` via `async_stream` + `tokio::interval`.
8. Implement `SubscribeEvents` via `broadcast::Receiver` with lag handling.
9. Streaming integration tests (with `tokio::time::pause`).

### Session 3 (V2c, ~1-2 days)

10. Add config fields + default helpers.
11. TLS identity helper (cert gen, load, validation).
12. Auth interceptor + token helper.
13. External-bind safety check (fail startup if external without TLS+auth).
14. Tests for each gate.

## Non-goals

- **gRPC-web proxy**: browser clients can't speak gRPC directly. Defer to explicit v3 (would require tonic-web middleware).
- **Reflection service**: useful for `grpcurl` but adds binary weight. Defer.
- **Mutations (WriteXxx)**: v2 is read-only. Writes require careful auth+authz design, deferred.
