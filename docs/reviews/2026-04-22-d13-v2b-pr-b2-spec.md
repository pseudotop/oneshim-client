## 2. Goals & Non-goals

### Goals

1. `SubscribeMetrics` yields a stream of `SubscribeMetricsResponse` items. First yield is always a `ServerLoadHint`. Subsequent yields are `MetricBucket` data (interleaved with transition/heartbeat `ServerLoadHint`).
2. Two cadence modes:
   - **Realtime** (`interval_secs == 0`): wake on `event_tx::Metrics` tick, coalesce within a 10ms quiet window, clamp to effective-interval floor.
   - **Interval** (`interval_secs > 0`): fire on `tokio::time::sleep(effective_interval)` schedule.
3. Load classification (`LoadPolicy`) drives enforcement. Four levels (LOW/MEDIUM/HIGH/CRITICAL) mapped from CPU% + free-memory-GiB, with 30s warm-up = MEDIUM.
4. Opt-out (`respect_server_hints = false`) honored iff trusted — loopback binding OR `Authorization: Bearer <integration_auth_token>` match. Otherwise silently downgraded + warn-logged.
5. Runtime kill switch: `WebConfig.grpc_streaming_enabled = false` → `SubscribeMetrics` returns `Status::unimplemented`. Unary v1/v2a RPCs unaffected.
6. All 10 existing v2a integration tests continue to pass.

### Non-goals

- `SubscribeEvents` implementation (PR-B3)
- Any new proto message or RPC beyond what PR-B1 already shipped (proto surface is frozen for PR-B2)
- Cross-subscriber DB query coalescing (still v3 backlog per design §Appendix B)
- TLS, dedicated gRPC auth token, external-bind (all v2c)
- `Ping` event emission (design explicitly out of scope)
- Storage-level changes (PR-B1 scope)

## 3. Wire contract (already-landed proto — no changes in PR-B2)

Quoted from `crates/oneshim-web/src/proto/...` (generated in PR-B1):

```proto
rpc SubscribeMetrics(SubscribeMetricsRequest) returns (stream SubscribeMetricsResponse);

message SubscribeMetricsRequest {
  uint32 interval_secs = 1;          // 0 = realtime, N = N-second bucket
  bool   respect_server_hints = 2;   // false = opt-out (trust-gated)
}

message SubscribeMetricsResponse {
  oneof payload {
    MetricBucket    data = 1;
    ServerLoadHint  hint = 2;
  }
}

message MetricBucket {                             // top-level, shared with v2a
  google.protobuf.Timestamp start = 1;
  double cpu_avg_pct    = 2;
  double memory_avg_mb  = 3;
  uint32 active_keystrokes   = 4;
  uint32 active_mouse_clicks = 5;
}

message ServerLoadHint {
  enum Level { LOAD_LEVEL_UNSPECIFIED=0; LOW=1; MEDIUM=2; HIGH=3; CRITICAL=4; }
  Level  load_level                 = 1;
  float  cpu_pct                    = 2;
  float  memory_pct                 = 3;
  uint32 suggested_interval_secs    = 4;
  uint32 suggested_event_rate_limit = 5;   // unused by SubscribeMetrics, set for parity
  string reason                     = 6;   // human-readable
  google.protobuf.Timestamp emitted_at = 7;
}
```

**Clamping & defaults**:

| Input | Server behavior |
|---|---|
| `interval_secs > 60` | Clamp to 60s ceiling (INTERVAL_CEILING) |
| `interval_secs == 0` | Realtime mode (floor 250ms INTERVAL_FLOOR) |
| `interval_secs` in (0, 1) requested as zero-second — not representable (uint32) | N/A — uint32 starts at 0 |
| `respect_server_hints = true` | Enforcement always on |
| `respect_server_hints = false` + trusted | Enforcement off (floor + ceiling still apply) |
| `respect_server_hints = false` + untrusted | Treated as `true`, warn-log once per stream |
| `grpc_streaming_enabled = false` | Immediate `Status::unimplemented` (no stream opened) |

### 4.1 `LoadPolicy`

Inputs:
- `thresholds: LoadThresholds` — new field on `WebConfig` (`grpc_load_thresholds: Option<LoadThresholds>`). `LoadThresholds { min_free_mem_gb, cpu_low_pct, cpu_medium_pct, cpu_high_pct }` with serde `#[serde(default = …)]` on each field. Default `(2.0, 50.0, 70.0, 90.0)`.
- `started_at: Instant` — set by `::new()`; warm-up window = first 30s.

Outputs:
- `classify(metrics: &SystemMetrics) -> LoadLevel`:
  - First 30s → `Medium` regardless of input (avoids sysinfo stabilisation noise).
  - Otherwise:
    - `Low` iff `cpu_pct < cpu_low_pct` AND `free_mem_gb > min_free_mem_gb * 1.5`
    - `Medium` iff `cpu_pct < cpu_medium_pct` AND `free_mem_gb > min_free_mem_gb`
    - `High` iff `cpu_pct < cpu_high_pct` AND `free_mem_gb > min_free_mem_gb * 0.5`
    - `Critical` otherwise (strict protective fallthrough — either signal alone can escalate)
- `enforced_metrics_interval(level, requested_secs) -> Duration`:
  - `requested = if requested_secs == 0 { INTERVAL_FLOOR=250ms } else { Duration::from_secs(requested_secs) }`
  - `level_floor = {Low: 250ms, Medium: 1s, High: 5s, Critical: 30s}`
  - Return `requested.max(level_floor).min(INTERVAL_CEILING=60s)`

`free_mem_gb` derivation: `(memory_total.saturating_sub(memory_used)) as f32 / 1_073_741_824.0`. `saturating_sub` guards the (currently unobserved) case where `memory_used > memory_total` — belt-and-braces, since sysinfo should never report that but panic on f32 subtraction underflow is unacceptable.

**Constructor validation**: `LoadPolicy::new` enforces `cpu_low < cpu_medium < cpu_high` AND `cpu_high <= 100.0` via `debug_assert!`. Violations are caller error (config validation happens earlier at config load). Tests cover only well-ordered thresholds.

### 4.2 `HintEmitter`

Per-stream state machine:

```
state = (last_level: Option<LoadLevel>, last_emit_at: Option<Instant>)

maybe_emit(level, policy, cpu_pct, memory_pct) -> Option<ServerLoadHint>:
  should_emit =
    last_level.is_none()                                 // first call
    OR last_level != Some(level)                         // transition
    OR last_emit_at.elapsed() >= HEARTBEAT (30s)        // heartbeat

  if should_emit:
    update state
    return Some(build_hint(level, policy, cpu_pct, memory_pct))
  else:
    return None

build_hint fields:
  load_level = protobuf enum mapped
  cpu_pct / memory_pct = passed through
  suggested_interval_secs = policy.enforced_metrics_interval(level, 0).as_secs() as u32
  suggested_event_rate_limit = policy.enforced_frame_rate(level) as u32  // set for parity
  reason = format!("{tag} (cpu={cpu:.1}% mem={mem:.1}%)")
  emitted_at = Utc::now() -> google.protobuf.Timestamp
```

Notes:
- Warm-up window is owned by `LoadPolicy` (level becomes MEDIUM). `HintEmitter` has no dedicated "warmup" reason string; `reason` simply reads `"MEDIUM (cpu=… mem=…)"` during warm-up. This is a deliberate simplification: the client infers warm-up from the stream being <30s old if it cares. The upstream design §3 mentioned a `"warmup"` prefix but folding it into `reason` inflates complexity with little observable benefit — **open question for spec review: keep simplified or add explicit warm-up reason tag?**

### 4.3 `auth_gate::honor_opt_out`

Pure function (no state, no async):

```rust
pub fn honor_opt_out(
    req_respect_hints: bool,
    remote_addr: Option<SocketAddr>,
    auth_header: Option<&str>,
    configured_token: Option<&str>,
) -> bool {
    if req_respect_hints { return true; }  // client opted in -> enforce
    if let Some(addr) = remote_addr {
        if addr.ip().is_loopback() { return false; }  // trust(a): loopback
    }
    if let (Some(h), Some(t)) = (auth_header, configured_token) {
        if h == format!("Bearer {t}") { return false; }  // trust(b): token match
    }
    true  // default: enforce
}
```

Returns **`true` when enforcement must stay ON** (opposite of "honor"ing opt-out — that bit of naming is legacy from the design doc, revisit naming in review).

**Token comparison**: plain `==`. v2b deliberately does not use constant-time comparison because:
- The token is presumed user-private and not attacker-probable over localhost.
- Logs never echo the token value.
- v2c will replace this with a dedicated token + TLS at which point constant-time comparison becomes meaningful.

**Open question for spec review**: is the `honor_opt_out` naming confusing? Returning `true`=enforce feels inverted for a function called "honor_opt_out". Alternatives: `must_enforce(...)`, `effective_respect(...)`. Spec review should settle.

### 4.6 `subscribe_metrics` generator (realtime + interval)

Pseudocode reflecting design §4 but tightened:

```text
subscribe_metrics(req):
  if !streaming_enabled: return Err(Status::unimplemented(...))

  remote_addr     = req.remote_addr()
  auth_header     = req.metadata().get("authorization").and_then(to_str)
  { interval_secs, respect_server_hints } = req.into_inner()

  enforcement_on = honor_opt_out(respect_server_hints, remote_addr, auth_header, token)
  if !respect_server_hints && enforcement_on:
      warn!("opt-out rejected; untrusted connection")

  let mut rx = event_tx.subscribe()
  let mut hint_emitter = HintEmitter::new()
  let mut last_emit: Option<Instant> = None

  return async_stream! {
    loop {
      // 1. Collect fresh metrics + classify
      metrics = system_monitor.collect_metrics().await?;
      level = load_policy.classify(&metrics);
      (cpu_pct, mem_pct) = derive_percents(&metrics);

      // 2. Maybe emit hint (initial / transition / heartbeat)
      if let Some(h) = hint_emitter.maybe_emit(level, &load_policy, cpu_pct, mem_pct):
        yield Ok(hint(h))

      // 3. Compute effective interval
      effective_interval = if enforcement_on:
          load_policy.enforced_metrics_interval(level, interval_secs)
        else:
          let r = if interval_secs == 0 { FLOOR } else { Duration::from_secs(interval_secs) };
          r.max(FLOOR).min(CEILING)

      // 4. Wait for next tick
      if interval_secs == 0:
        match rx.recv().await:
          Ok(RealtimeEvent::Metrics(_)) => wake
          Ok(_)                          => continue  // ignore other variants
          Err(Lagged(_))                 => continue  // metrics fires on schedule anyway
          Err(Closed)                    => return   // server shutdown
        # Coalesce additional wake-ups within 10ms quiet window
        while timeout(10ms, rx.recv()).await.is_ok() { /* drain */ }
        # Rate-limit: skip if within effective_interval since last_emit
        if let Some(t) = last_emit { if t.elapsed() < effective_interval { continue; } }
      else:
        tokio::time::sleep(effective_interval).await

      // 5. Fetch bucket
      window_start = Utc::now() - chrono::Duration::from_std(effective_interval).unwrap_or(seconds(1))
      window_end   = Utc::now()
      bucket = spawn_blocking(|| storage.aggregate_metrics_window(window_start, window_end)).await
      match bucket:
        Ok(Ok(b))   => yield Ok(data(b)); last_emit = Some(Instant::now())
        Ok(Err(e))  => warn!(err.code=%e.code(), "..."); continue    // transient DB — skip tick
        Err(join)   => yield Err(Status::internal(...)); return     // spawn_blocking panic — fatal
    }
  }
```

**Important semantic choices**:

1. **First iteration always emits a hint**: `HintEmitter` returns `Some(...)` on first call because `last_level = None`. Subsequent iterations only emit on transition / 30s heartbeat. **First data bucket** lands on the next tick after the first hint (realtime: wait for `event_tx` wake; interval: sleep(effective_interval)). There is NO immediate-data-on-subscription — the client sees `Hint → wait → Data`. An alternative ("fetch-and-emit initial data before the wait") is explicitly rejected: it would require backdating `window_start = Utc::now() - default_1s`, which would be inconsistent with the `effective_interval` derivation later. Simpler invariant: every data payload has `window_start = now - effective_interval`.

2. **Realtime coalescing happens AFTER the first `rx.recv()` that returned `Metrics`**, not before. Rationale: the first event that wakes us is already 1 monitor-tick old; draining subsequent queued wakeups cuts a burst of 3-4 ticks from 50ms latency down to the 10ms quiet window. The design doc pseudocode had the quiet-drain happening INSIDE the `recv()` retry loop — our version is cleaner because the single `Ok(Metrics)` branch gates coalescing.

3. **Transient DB errors `continue`**: the next iteration will retry. Consecutive failures do NOT get counted — the stream stays open indefinitely unless `Err(Closed)` or `spawn_blocking` panic fires. Design §5 mentioned "on N=10 consecutive failures close stream", but this adds state without clear benefit: the client's heartbeat timeout will already close stale streams, and Prometheus `dashboard.db_query_duration` + error-logs catch the condition. **Open question**: revisit in review — add consecutive-failure counter or accept always-retry?

4. **`spawn_blocking` JoinError is fatal**: `tokio::task::JoinError` happens only when the blocking task panicked or was cancelled. Either way the stream can't recover from a half-executed DB query. We yield `Status::internal` once and `return` to drop the generator, releasing `rx`.

5. **SystemMonitor failure (`collect_metrics() -> Err`)**: per design §5 should use last-good for up-to 5s, then lock to MEDIUM with `reason="metrics_stale"`. **PR-B2 simplification**: we yield `Status::internal` and exit. Rationale: metrics collection is synchronous sysinfo refresh; never observed to fail in production since 2024. Implementing last-good caching adds ~40 LoC of state for a pathological case. **Open question for spec review**: is the simplification acceptable or must we implement 5s stale tolerance?

## 6. Security & privacy posture for PR-B2

Inheriting design §6.1:

| Concern | PR-B2 posture |
|---|---|
| Wire confidentiality | Loopback-only bind inherited from v1/v2a (v2c flips the gate) — no network-level leak |
| PII on wire | `MetricBucket` fields are all aggregated numerics; `ServerLoadHint` is server-authored numeric + enum + reason. **Zero PII paths in PR-B2 payloads.** The `pii_sanitizer` field in `GrpcSpawnConfig` is placeholder for PR-B3 AiRuntimeStatus.fallback_reason |
| Opt-out trust | T2-extended (loopback OR matching token). Logging **never** echoes the token value — only `present/absent` markers |
| Token leakage via logs | Unit test (new, in PR-B2): instrument `tracing` with a test subscriber, assert the literal token substring does not appear in any emitted log during a full opt-out attempt |
| DoS via many subscribers | Design §5 mandates `active_stream_counter: AtomicUsize` with cap 50 across both RPCs. **PR-B2 scope: add counter + cap, test it with 51 concurrent connects** (new unit-level integration test). Counter increments on subscribe entry, decrements on generator drop. Enforced via `Status::resource_exhausted` |
| Malformed `Authorization` header | `honor_opt_out` treats it as no-match → enforcement stays on. Test `opt_out_rejected_malformed_auth_header` covers |

|---|---|---|---|
| `subscribe_metrics` realtime path races with `event_tx` capacity (256) under many subscribers | Low (PR-B2 adds 1 sub/stream × max 50 streams = 50 subs vs 256 capacity) | Medium (Lagged drops = stream noise) | `DroppedEventsSignal` is PR-B3-only; PR-B2 simply `continue`s on `RecvError::Lagged(_)` — metrics tick is scheduled so no data is actually lost |

## 12. Glossary

- **Effective interval** — `load_policy.enforced_metrics_interval(level, requested_secs)` — after enforcement ladder + floor/ceiling clamps
- **Warm-up window** — first 30s after `LoadPolicy::new()`; always classified as MEDIUM
- **Opt-out** — `respect_server_hints = false` on the request; only honored when trusted (loopback / token match)
- **Trusted connection** — loopback address OR matching configured `integration_auth_token`
- **INTERVAL_FLOOR** — `Duration::from_millis(250)` — absolute lower bound on emit cadence
- **INTERVAL_CEILING** — `Duration::from_secs(60)` — absolute upper bound on emit cadence
- **HEARTBEAT** — `Duration::from_secs(30)` — hint emitter's heartbeat interval
- **`DashboardEventSignal`** — internal enum `{ Frame(i64), Idle, AiRuntimeStatus }` landed in PR-B1; only `Frame` carries an id (other two are "latest-state" lookups); PR-B2 does not construct any signal (SubscribeEvents only)
