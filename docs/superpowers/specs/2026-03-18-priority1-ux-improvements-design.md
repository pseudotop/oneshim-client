# Priority 1: UX Improvements — Design Spec

> Created: 2026-03-18
> Status: Implemented
> Depends on: Layer 2 (LLM Summary + Vector RAG, merged)

## 1. Goal

Deliver three user-facing features that make the analysis pipeline's value tangible: a timetable-based regime visualization dashboard, hybrid vector+keyword search, and daily digests with LLM feedback.

## 2. Design Decisions

| Item | Decision | Rationale |
|------|----------|-----------|
| Dashboard placement | New tab/section in existing React frontend (oneshim-web) | Already serves as client GUI |
| Dashboard layout | Timetable view (top) + statistics (bottom) | TickTick/Google Calendar style time blocks |
| LLM feedback display | Top insight card + inline timeline annotations | Most important element — coach-like presence |
| Search strategy | Hybrid: `α·vector_score + β·fts5_score` | Best relevance without user mode switching |
| Daily digest format | Timetable-based summary with LLM narrative | More informative than raw numbers |
| API pattern | REST endpoints in oneshim-web + Tauri IPC commands | Consistent with existing architecture |

#### LLM daily insight generation

Triggered at end of day (or on-demand) via the scheduler. Uses `AnalysisProvider::summarize_text()` with a daily insight prompt that receives:
- All segments for the day (from `activity_segments`)
- Content activities with work types
- Regime transitions
- Comparison with previous day

The prompt instructs the LLM to generate:
1. A 2-3 sentence narrative summary
2. Inline annotations for notable segments (achievements, warnings, suggestions)

#### Tauri command

```rust
#[command]
pub async fn get_dashboard_day(date: String) -> Result<DashboardDayResponse, String>
```

### 3.4 Data assembly

The dashboard endpoint assembles data from existing tables:
- `activity_segments` columns: `start_time`, `end_time`, `duration_secs`, `regime_id`, `trigger_reason`, `llm_summary`, plus JSON columns: `app_breakdown`, `content_activities_json`, `category_breakdown`, `patterns_json`, `container_json`
- `regimes` → regime labels and colors
- Previous day's `daily_digests` → statistics delta comparison

**Caching**: On first request for a date, the dashboard generates the view from `activity_segments`, calls LLM for insight if enabled, and stores the result in `daily_digests` table (V11). Subsequent requests serve from cache. The `daily_digests` table is both a cache and the daily digest API's data source (§5.6).

**Graceful degradation**: If LLM is unavailable, `insight` is `null` in the response. The frontend renders a "No insight available" placeholder. Empty days return an empty timeline with zeroed statistics.

## 4. Feature 2: Hybrid Search (Vector + FTS5)

### 4.1 Architecture

```
User query: "auth module JWT"
       ↓
┌─────────────────────────────────────┐
│  HybridSearchService                │
│                                     │
│  ┌───────────┐  ┌───────────────┐  │
│  │ Vector    │  │ FTS5 Keyword  │  │
│  │ Search    │  │ Search        │  │
│  │ (embed →  │  │ (SQL MATCH    │  │
│  │  KNN)     │  │  query)       │  │
│  └─────┬─────┘  └──────┬────────┘  │
│        │               │           │
│        ▼               ▼           │
│  Reciprocal Rank Fusion (RRF)      │
│  score = α/(k+rank_v) + β/(k+rank_f)│
│        │                           │
│        ▼                           │
│  Merged + deduplicated results     │
└─────────────────────────────────────┘
```

### 4.2 FTS5 Setup

Create FTS5 virtual table (V11 migration) indexing:
- `activity_segments.stats_json` → extracted text (app names, content labels)
- `activity_segments.llm_summary`
- `embedding_vectors.original_text`

```sql
CREATE VIRTUAL TABLE search_fts USING fts5(
    segment_id UNINDEXED,
    content_type,           -- 'segment' | 'content_activity'
    searchable_text,        -- extracted text for keyword matching
    tokenize='porter unicode61'
);
```

Populated by trigger or on INSERT to `activity_segments` and `embedding_vectors`.

### 4.3 Reciprocal Rank Fusion (RRF)

Better than linear score combination because vector similarity (0-1) and FTS5 scores (BM25) are on different scales. RRF uses rank position instead:

```
rrf_score(d) = α / (k + rank_vector(d)) + β / (k + rank_fts5(d))
```

Where:
- `k = 60` (standard RRF constant)
- `α = 0.6` (vector weight — semantic understanding)
- `β = 0.4` (FTS5 weight — exact keyword matching)
- If a document appears in only one result set, the missing rank = `limit + 1`

### 4.4 API

Extend existing `GET /api/semantic-search` (avoids collision with pre-existing `/api/search`):

```
GET /api/semantic-search?q=auth+module+JWT&mode=hybrid&limit=10
```

Query parameter `mode`:
- `hybrid` (default): vector + FTS5 RRF fusion
- `semantic`: vector-only (current behavior)
- `keyword`: FTS5-only

Response format unchanged (`Vec<EnrichedSearchResult>`).

### 4.5 Dependencies

Add `"fts5"` feature to `rusqlite` in workspace Cargo.toml:
```toml
rusqlite = { version = "0.38", features = ["bundled", "fallible_uint", "fts5"] }
```

FTS5 index is populated at application level (not via SQLite triggers — consistent with existing codebase pattern). On segment INSERT, also INSERT into `search_fts` in the same transaction.

Pre-V11 data backfill: the V11 migration includes a backfill query to index existing segments.

### 4.6 Backend changes

- New `FtsSearchService` in `oneshim-storage` — manages FTS5 index sync + MATCH queries
- Modified `SemanticSearchHandler` — calls both search paths, merges via RRF
- FTS5 index populated on segment/embedding insert (sync, lightweight)

## 5. Feature 3: Daily Digest

### 5.1 Format — Timetable-based summary

The daily digest is a structured document combining:
1. **LLM narrative** — top insight card content
2. **Timetable** — segment list with times, regimes, content
3. **Inline LLM annotations** — highlights on notable segments
4. **Statistics** — summary numbers with previous day comparison

### 5.2 Data model

```rust
pub struct DailyDigest {
    pub date: chrono::NaiveDate,
    pub insight: DailyInsight,
    pub timeline: Vec<TimelineEntry>,
    pub statistics: DailyStatistics,
    pub generated_at: DateTime<Utc>,
}

pub struct DailyInsight {
    pub narrative: String,
    pub highlights: Vec<DigestHighlight>,
}

pub struct DigestHighlight {
    pub highlight_type: HighlightType,  // Achievement, Warning, Suggestion
    pub text: String,
    pub segment_id: Option<String>,
}

pub struct TimelineEntry {
    pub segment_id: String,
    pub start_time: DateTime<Utc>,
    pub end_time: DateTime<Utc>,
    pub duration_mins: u32,
    pub regime_label: String,
    pub dominant_app: String,
    pub content_summary: Vec<ContentBrief>,
    pub annotation: Option<DigestHighlight>,
}

pub struct ContentBrief {
    pub content: String,
    pub work_type: WorkType,
    pub mins: u32,
}

pub struct DailyStatistics {
    pub deep_work_hours: f32,
    pub communication_hours: f32,
    pub meeting_hours: f32,
    pub context_switches: u32,
    pub longest_focus_mins: u32,
    pub longest_focus_content: String,
    pub regime_distribution: HashMap<String, u32>,  // percentage
    pub comparison: Option<DayComparison>,
}

pub struct DayComparison {
    pub deep_work_delta: f32,
    pub communication_delta: f32,
    pub context_switch_delta: i32,
}
```

### 5.3 Generation

#### Automatic trigger

In the scheduler aggregation loop, check at midnight (configurable):
- Query `activity_segments` for today
- Build `DailyDigest` from segments (pure aggregation, like `WeeklyDigestGenerator`)
- If LLM enabled, generate insight narrative + inline annotations
- Store in new `daily_digests` table

#### On-demand

```
GET /api/digests/daily?date=2026-03-18
GET /api/digests/daily/today
```

Tauri command: `get_daily_digest(date: Option<String>)`

### 5.4 LLM Prompt for Daily Insight

```
You are a productivity coach analyzing a day of desktop work activity.

Given the day's activity data (segments with apps, content, work types, durations),
provide:
1. A 2-3 sentence narrative summary of the day
2. Up to 5 highlights — achievements (🏆), warnings (⚠️), or suggestions (💡)
   Each highlight references a specific segment or pattern.

Respond in JSON:
{
  "narrative": "...",
  "highlights": [
    {"type": "achievement", "text": "...", "segment_id": "seg-42"},
    {"type": "warning", "text": "...", "segment_id": "seg-43"},
    {"type": "suggestion", "text": "..."}
  ]
}
```

### 5.5 LLM Response Parsing & Fallback

The LLM may return JSON wrapped in markdown fences, with preamble, or malformed. The `DailyInsightGenerator` must:

1. **Extract JSON**: Strip markdown fences (` ```json ... ``` `), find first `{` to last `}`, parse as `DailyInsight`
2. **Validate**: Check `narrative` is non-empty, `highlights` is an array
3. **Fallback on parse failure**: Generate a basic insight from statistics only:
   ```rust
   DailyInsight {
       narrative: format!("오늘 딥워크 {:.1}시간, 커뮤니케이션 {:.1}시간.", dw_hours, comm_hours),
       highlights: vec![],
   }
   ```
4. **Fallback on LLM unavailable**: `insight` is `None` in the response, frontend shows placeholder

This reuses the markdown fence stripping logic already in `AnalysisClient::parse_candidates()`.

### 5.6 Storage

V11 migration adds:

```sql
CREATE TABLE daily_digests (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    date TEXT NOT NULL UNIQUE,
    insight_json TEXT,        -- DailyInsight serialized
    timeline_json TEXT NOT NULL,
    statistics_json TEXT NOT NULL,
    generated_at TEXT NOT NULL DEFAULT (datetime('now'))
);
-- No separate index needed: UNIQUE constraint on `date` creates implicit index
```

### 5.6 Dashboard integration

The daily digest IS the dashboard's data source. The `GET /api/dashboard/day` endpoint returns the same data as the daily digest, formatted for the frontend. If a digest exists, serve it. If not, generate on-the-fly (and cache).

## 7. FTS5 Search as Port Trait

To avoid `oneshim-analysis` depending on `oneshim-storage` (hexagonal violation), FTS5 search is exposed as a port:

```rust
// oneshim-core/src/ports/text_search.rs
#[async_trait]
pub trait TextSearchProvider: Send + Sync {
    async fn search_fts(&self, query: &str, limit: usize) -> Result<Vec<TextSearchResult>, CoreError>;
    fn sync_segment(&self, segment_id: &str, searchable_text: &str) -> Result<(), CoreError>;
}

pub struct TextSearchResult {
    pub segment_id: String,
    pub content_type: String,
    pub matched_text: String,
    pub rank: f32,  // BM25 rank from FTS5
}
```

`HybridSearchService` in `oneshim-analysis` accepts `Arc<dyn TextSearchProvider>` + `Arc<dyn VectorStore>` — both are port traits from `oneshim-core`. No adapter-to-adapter dependency.

## 8. Phase Scope

### This implementation

1. `DailyDigest` + related models in `oneshim-core`
2. `DailyDigestGenerator` — segment aggregation
3. `DailyInsightGenerator` — LLM narrative + highlights
4. V11 migration (FTS5 + daily_digests)
5. FTS5 index sync on segment/embedding insert
6. `HybridSearchService` — RRF fusion
7. `GET /api/dashboard/day` — timetable + insight + stats
8. `GET /api/digests/daily` — daily digest API
9. Tauri commands for dashboard + digest
10. Scheduler hook for daily digest auto-generation
11. React frontend: TimelineView, InsightCard, StatisticsPanel, SearchBar

### Out of scope
- GUI activity intelligence (OCR + mouse correlation) — separate feature
- Multi-day comparison view (beyond prev-day delta)
- Custom regime color themes
- Export daily digest as PDF/image
